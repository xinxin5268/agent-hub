## 二、融合方案

### 核心思路：不部署外部系统，而是让 Agent Hub 的 Agent 集群自己长出写作能力

```
┌─────────────────────────────────────────────────────────────┐
│                    用户创作指令                             │
│  "写一本修仙文，主角是重生者，要有金手指和逆袭爽点"          │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│                    小宝 (CEO) — 决策层                      │
│  拆解需求 → 选题材Skill → 分配Agent团队 → 监控进度 → 审核  │
│  参考: Toonflow 的 script_agent_decision.md                │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│                    执行层 Agent 集群                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Agent 1: 叙事学家 (世界观架构师)                          │
│  → 构建世界规则、力量体系、阵营格局                         │
│  → 参考: AI-Novel-Writing-Assistant 的世界管理 v2         │
│  → 产出: world.yaml (分层设定 + 约束)                      │
│                                                             │
│  Agent 2: 心理学家 + 叙事设计师 (角色系统)                 │
│  → 设计角色人格、动机、关系图谱、弧线                       │

│  Agent 3: 内容创作者 + 图书联合作者 (大纲+拆章)            │
│  → 生成卷战略、节奏板、章节蓝图                             │

                         ↓

│                    监督层 Agent                             │
│  叙事学家: 结构一致性 + 伏笔回收                           │
│  心理学家: 角色行为一致性                                   │
```

## 三、各 Agent 能力打磨计划

### Phase 1: 给 Agent 注入写作能力定义（核心层）

每个 Agent 需要新增的创作能力声明（写入 agents.json）：

```json
{
  "id": "xiaobao",
  "skills": [
    "ceo-agent",
    // 新增:
    "novel-direction",      // 小说创作决策
    "agent-orchestration",  // 创作Agent编排
    "quality-gate"          // 门禁审核
  ]
}
```

**小宝** 新增能力：
- `novel-direction` — 从用户一句话拆解出创作需求，选题材、定方向
- `agent-orchestration` — 分配执行Agent、监控进度、中断恢复
- `quality-gate` — 最终产出审核

**小聪** 新增能力：
- `world-building` — 世界观构建（分层设定+约束）
- `character-system` — 角色系统（15维事实快照）
- `chapter-review` — 章节审核（连贯性+字数）

**OpenCode WSL** 新增能力：
- `writing-formula` — 写法引擎（从文本提取写法特征）
- `outline-generation` — 大纲生成（卷战略+拆章）
- `batch-production` — 批量章节生产
- `anti-ai-detection` — AI味检测+润色

**OpenCode Win** 新增能力：
- `creative-hub-ui` — 写作UI（参考91Writing的编辑器）
- `card-system` — 卡片式创作界面
- `knowledge-graph` — 知识图谱可视化

**千问A3** 新增能力：
- `genre-research` — 题材调研（分析热门网文套路）
- `fact-check` — 事实一致性校验
- `style-analysis` — 写法分析（拆书+提取风格）

---

### Phase 2: 题材Skill体系（参考Toonflow 12种导演skill）

每个题材作为独立 skill 文件，Agent 按需加载：

```
skills/novel-genre/
├── xianxia.md          # 玄幻修仙 — 力量体系/境界划分/丹道
├── urban.md            # 都市 — 职场/商战/豪门
├── sci-fi.md           # 科幻 — 硬科幻/赛博/末世
├── wuxia.md            # 武侠 — 内力/招式/江湖
├── romance.md          # 言情 — 甜宠/虐恋/破镜重圆
├── suspense.md         # 悬疑 — 推理/密室/连环案
├── history.md          # 历史架空 — 权谋/争霸/宫斗
├── horror.md           # 恐怖 — 灵异/克苏鲁/民俗
├── game.md             # 游戏/无限流 — 副本/升级/技能
├── fan-fiction.md      # 同人 — 原作设定继承
└── hybrid.md           # 多题材融合规则
```

每个 skill 包含：
- 题材核心设定模板（世界观/力量体系/常见套路）
- 写法规则（节奏/爽点/悬念密度）
- 反AI特征库（该题材常见AI味）
- 平台审核要点

### Phase 3: 一致性系统（参考天命6道门禁）

在 Agent Hub 的知识共享层实现：

```
写作流程 → 6道门禁校验 → 通过才落地
            ↓
         ┌──────────────────────────────┐
         │  1. 协议解析                   │
         │     AI输出必须含 CHANGES 声明  │
         ├──────────────────────────────┤
         │  2. 引用校验                   │
         │     引用的角色/地点ID必须存在  │
         ├──────────────────────────────┤
         │  3. 一致性校验                 │
         │     角色状态不矛盾（张三畏火） │
         ├──────────────────────────────┤
         │  4. 未知实体检测               │
         │     新实体数量控制             │
         ├──────────────────────────────┤
         │  5. 描写一致性                 │
         │     发色/瞳色/服装一致         │
         ├──────────────────────────────┤
         │  6. 蓝图出场检查               │
         │     关键角色必须出场           │
         └──────────────────────────────┘
```

状态回写机制：
```
每章完成后 → 更新 facts.jsonl（15维事实快照）
             → 更新伏笔状态（已埋/已收/未收）
             → 更新角色位置
             → 更新势力关系
```

### Phase 4: 写法引擎（参考AI-Novel-Writing-Assistant）

Agent 之间通过知识库共享写法资产：

```
写法来源:
  ├─ 拆书分析（千问A3 → 提取写法特征）
  ├─ 用户手动创建
  ├─ 粘贴范文提取
  └─ 当前作品反推

写法资产结构:
  ├─ 描述层（给用户看的自然语言）
  └─ 执行层（结构化规则 → 注入生成prompt）

反AI特征库:
  ├─ 总结句检测: "就这样" "总之" "不得不说"
  ├─ 升华句检测: "也许这就是人生"
  ├─ 解释性心理描写: "他心想..."
  ├─ 对称句式: "不是因为...而是因为..."
  └─ 通用感叹: "命运的车轮开始转动"
```

---

### Phase 5: 知识库回灌

```
每章生成后 →
  1. 提取关键事件 → 写入 knowledge.json (type: plot)
  2. 提取角色状态变更 → 写入 knowledge.json (type: character)
  3. 提取新设定 → 写入 knowledge.json (type: world)
  4. 提取写法特征 → 更新写法引擎
  5. 提取未回收伏笔 → 追踪到伏笔列表
```

## 四、实施步骤

### Step 1: 定义 Agent 写作能力（1天）
- 给每个 Agent 新增 skill 声明
- 在 agents.json 注册

### Step 2: 创建题材Skill模板（2-3天）
- 先做 3 个核心题材：玄幻/都市/悬疑
- 其余逐步添加

### Step 3: 实现一致性门禁（2天）
- 写一个校验脚本（参考天命的CHANGES协议）
- 接入每章生成流程

### Step 4: 实现状态回写（1天）
- facts.jsonl 格式定义
- 每章后自动更新

### Step 5: 实现写法引擎（2天）
- 从拆书提取写法特征
- 反AI特征库
- 写法绑定到项目

### Step 6: 整合测试（1天）
- 跑一轮完整流程：用户指令→Agent协作→小说产出→门禁→回写

## 六、评估

| 目标 | 实现方式 | 可信度 |
|------|---------|--------|
| 长篇一致性 | 天命门禁 + 事实快照 + 每章状态回写 | 🔒 已验证可行 |

| 平台过审 | 反AI特征库 + 多轮润色 + 题材适配 | ⚠️ 需要实测调整 |
| 创造性 | 写法引擎约束框架 + 多Agent头脑风暴 + 人机协作 | 🔄 持续优化 |
